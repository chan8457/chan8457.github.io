## 프로젝트 개요

서비스가 종료된 10대 대상 익명 투표 앱의 실제 로그 데이터를 받아,  
**“회사가 이 서비스를 새롭게 만들어 내라고 했다”** 는 시나리오로 재런칭 전략을 수립한 프로젝트입니다.  
핵심 목표는 두 가지였습니다.

1. 왜 피크 대비 **DAU가 97%나 급락했는가**  
2. 다시 출시한다면 **어디부터 고쳐야 유지가 되는가**  

앱의 특성상 “친구가 있어야 재미가 나는 구조”인데, 첫 24시간 안에 핵심 기능(질문/투표)을 경험하지 못해 정착에 실패한 사용자가 대다수라는 걸 확인했습니다. :contentReference[oaicite:1]{index=1}

| 항목 | 내용 |
|------|------|
| 기간 | 2024.07 ~ 2024.08 |
| 인원 | 팀 프로젝트 (4명) |
| 역할 | 이벤트 로그 모델링, 코호트/리텐션 분석, 이탈 예측 피처 설계, 파이프라인 설계 |
| 기술 | Python, SQL(BigQuery), Airflow, BQML, Dashboard, Discord Webhook |
| 데이터 | 가입/로그인/질문 생성/투표/친구 초대/미복귀 로그 전체 |

## 문제 정의

- 피크 DAU 316,872 → 7,429로 **97.7% 급락**
- 신규 가입자의 **70%가 핵심 기능(질문·투표)을 한 번도 못 경험**
- 첫 주 리텐션 44% 수준 → 커뮤니티 서비스로는 치명적
- ‘친구가 있어야 재미있다’는 구조인데 **핵심 사용자를 중심으로 네트워크가 형성되지 못함**  
  → 그래서 “누가 중심이 될 수 있는가”와 “누가 곧 나갈 사람인가”를 동시에 보는 게 필요했음. :contentReference[oaicite:2]{index=2}

## 분석 구조

### 1️⃣ 퍼널·리텐션 분석
- 온보딩 후 24시간 안에 **첫 투표 or 첫 질문 생성**을 하면 리텐션이 유의하게 높아짐
- 반대로 “구경만 하고 떠난 유저”는 7일 이내 미복귀 비율이 매우 높음
- 그래서 **“첫 세션 내 핵심 경험 강제 노출”**을 최우선 개선안으로 설정

### 2️⃣ 핵심 유저(중심 학생) 찾기
- 10대 서비스는 학교·학급 단위로 묶이는 경향이 커서,  
  활동량 높은 몇 명이 전체 방식을 결정함
- 친구 초대수, 받은 투표수, 질문 생성수로 스코어링해서 **코어 유저를 먼저 케어하는 운영 플로우** 설계
- 이 유저가 이탈하면 그 뒤에 붙은 유저들도 같이 빠져나가므로 알림/혜택을 이 그룹에 우선 제공

### 3️⃣ 조기 이탈 예측 모델
- “향후 7일 미복귀”를 타깃으로 하는 이탈 예측 모델 설계
- 주요 피처 예시:
  - 첫 1일 동안의 접속 횟수
  - 첫 1일 동안의 투표/질문 참여 여부
  - 친구초대 / 받은 투표 수
  - 새 알림 클릭 여부
- 예측 결과를 **Discord로 운영자에게 쏘는 플로우**까지 설계  
  → 운영자가 “이탈 위험 높은 애들”만 골라 푸시/쿠폰/이벤트를 줄 수 있게 함  
  → 파이프라인: **Airflow → GCS → BigQuery (bronze/silver/gold) → BQML → Dashboard → Discord**  
  → 일 단위로 자동화 목표. :contentReference[oaicite:3]{index=3}

## 개선안

| 구분 | 문제 | 개선 아이디어 |
|------|------|----------------|
| 첫 24시간 정착 실패 | 핵심 기능 노출 이전 이탈 | 첫 세션에서 바로 “내 학교/내 반 투표 보기” 진입 버튼 노출 |
| 친구 네트워크 약함 | 활동 많은 학생 보호 필요 | 중심 학생 주간 랭킹 만들고 이탈 위험 시 운영 알림 |
| 일괄 이벤트 운영 비효율 | 모든 유저에게 푸시 → 반응 낮음 | 이탈 예측 모델로 위험군만 타깃팅 |
| 앱 재방문 낮음 | 투표 결과를 나중에 보게 되는 구조 | 투표 결과 알림/댓글 알림을 푸시로 전환 |

## 기대 효과

- “첫날 핵심 경험 노출률”을 올리면 자연스럽게 7일 리텐션이 따라 올라감
- 운영이 모든 유저를 케어하는 게 아니라 **“지금 붙잡아야 할 유저”** 위주로 움직일 수 있음
- 데이터 파이프라인을 깔아놨기 때문에 이후 추천/랭킹/피드 정렬에도 재사용 가능
